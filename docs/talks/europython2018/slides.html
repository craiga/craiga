Hello!

I'm Craig Anderson, I'm a web developer, and I wanted to share my experience of going to EuroPython this year.

It's a big conference and this is a short talk so I can't go into too much detail.

Hopefully it'll give you a few pointers to things to look into.

Overview

  • 17th EuroPython

This was the 17th EuroPython, and the first in Edinburgh.

More than 1000 attendees.

Event went for a week.

Started on Monday and Tuesday with workshops.

Wednesday, Thursday and Friday were the main conference days.

Saturday and Sunday were sprints, which are hackathon-like days to help people make contributions to open source.

Live streamed on YouTube. Links to all talks on web.

Given that this talk is all about my experience, so I thought I should introduce myself.

I consider myself more of a web developer than a Python.

I'm not from this part of the world.

This was my only my second open source conference. I went to DjangoCon a few months back and really loved it. I wrote some blog posts about that which you can take a look at.

That experience, combined wanting to visit the country my family came from, was my main motivation.

I didn't feel like a three star Python developer.

Going to something like this, you really need to mindful when impostor syndrome starts kicking in.

They get you to rank your Python experience. 3 stars = intermediate, which felt about right. Met lots of other people who seemed to know way more than I did who were also three stars. I'm not sure what the point of this was.

The first two days were filled with half-day workshops, of which I attended two.

To be honest, not really worthwhile unless there's a lot you're specifically interested in.

Probably good for beginners.

My first workshop was about Pandas and NumPy.

I've used Pandas before but didn't feel like I really get it, so this was valuable.

I get the impression that Pandas is quite difficult to learn and its API is volatile.

There are a lot of non-obvious best practices about when to use nulls and when to use not a number, which I found a bit confusing, but it's good to know that it's a thing.

I've wanted to get into AWS Lambda for a while, so this was a good intro.

Was all about Chalice, Amazon's own Lambda serverless framework, and integration with other AWS services.

Lots of "here's one I made earlier" copying and pasting of CloudFormation templates.

Discussed Zappa, a competing open source framework with another attendee.

Also heard about Cloud Tools, a collection of open-source tools to do things like create Cloud Formation templates in Python.

The workshop instructions are available online.

So, on to the main bit of the conference.

I really loved the non-technical talks‚Äîtalks about software development as a craft, how to be more inclusive, and how we fit into society at large.

It might be a bit weird how much I love them.

Daniele has given this talk at DjangoCon, Pylondinium and now EuroPython. It's got nothing to do with Python and everything to do with how we approach our craft and deal with amateurs. Highly recommended. Daniele is an excellent speaker.

Ines gave a keynote that cut through a lot of the conventional wisdom that's out there about how to build a business on the internet. Advised against low-value, winner-takes-all spaces.

Introduced me to the idea of mittelstand (probably botched the pronunciation of that), a class of German, family-owned, middle-sized, middle class business. They're not looking to take over the world, but do the best work in their little area.

Keith runs a software crafters meetup in Glasgow called CodeCraft. He describes it a guided conversation between software developers. Sounds like an interesting idea and something I'd be very keen to see brought down to London.

He also spoke about ownership and authorship and how that can promote good cultures in development teams.

Ed's a London-based developer who's recently been diagnosed with autism, and it was very interesting to hear how that diagnosis has helped him. Also offered lots of concrete advice for making workplaces better for those suffering from autism, which all sounded like great advice anyway: plan work pipelines using Kanban or similar, manage discussions, workplaces with minimal distractions, etc.

Josh's talk taught me about how Python fits into the UK's education system. He's built a product called Edublocks, which marries the simple drag and drop programming environment Scratch (which is what young children use to learn about the basic concepts of programming) with the power of Python.

Also, he's only 14 years of age. He was a great speaker, and I look forward to seeing what else he brings to Python, education, and the world at large.

Ed and Josh's talk actually inspired me to reach out to my sister and her seven year old child who is autistic. I've brought him some toys designed to help teach programming for his upcoming birthday, and I really hope he likes them.

Technical Talks

Python 2 is Dead! Drag Your old Code Into the Modern Age

Becky Smith

  • caniusepython3
  • 2to3
  • pylint --py3k
  • future, six, modernise
  • tox
  • python3porting.com

Moving onto some of the more technically detailed talks.

The subject of Becky's talk is super-relevant to my current client as Python 2 will not be maintained after January 1, 2020.

She shared her experience of porting a 65 thousand line Django app from Python 2 to 3 using tools such as caniusepython3 to test dependencies, 2to3 to automate a lot of the conversion, and pylint to detect Python 3 issues which might not be immediately obvious.

She also discussed writing code which runs across both versions using future, six and modernise, as well as tox for automating testing across Python versions.

Did you know that Flask was started as a joke? Apparently somebody thought that Flask embodied everything's that wrong about microframeworks.

For example, hug is a bit like Flask, but designed very specifically for REST APIs.

At the end of each day of talks there were a series of lightning talks, which are little five minute talks people can sign up ti give on the day.

These can be some of the most informative bits of the conference!

One of the first made the case that the Zen of Python can be harmful when taken as gospel.

It was initially written with tongue in cheek, and we should remember that when citing it.

I've only ever copied code from StackOverflow to make my own decorators, so I had no idea how much complexity there is in this area. There were a lot of talks about decorators, which was a new thing to me.
